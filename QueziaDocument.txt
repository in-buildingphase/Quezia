Quezia is an exam-agnostic, AI-powered mock testing and performance intelligence platform built around strict structural separation, versioned test orchestration, deterministic grading, and materialized analytics. It is not a question viewer, not a content repository, and not a chat interface. It is a structured assessment engine designed to simulate real exams with fidelity while allowing flexible, user-driven practice configurations.
The backend architecture must reflect this identity through layered responsibility separation and immutability where required.
________________


FOUNDATIONAL LAYER: IDENTITY AND ACCESS CONTROL
The system begins with role-based user management.
A User entity must contain:
* id

* email

* username

* passwordHash

* role (LEARNER or ADMIN)

* isActive

* createdAt

* lastLogin

* USER PROFILE LAYER (ONBOARDING & PERSONAL CONTEXT)
In addition to the core User entity used for authentication and authorization, Quezia must include a dedicated UserProfile layer. This layer is responsible for storing personal, academic, and onboarding-related information that shapes the learner’s experience, personalization, analytics context, and future product expansion.
The User table should remain minimal and security-focused (email, password, role, status). The UserProfile table must handle all contextual and personalization data.
________________


Purpose of UserProfile
The UserProfile serves four primary functions:
   1. Onboarding Context
Captures essential information during the onboarding flow to personalize the platform experience.

   2. Academic Configuration
Stores exam preferences, target year, and preparation stage.

   3. Personalization Layer
Enables dynamic UI behavior, AI recommendations, and tailored analytics.

   4. Future Expandability
Allows storing long-term learner attributes without polluting the core authentication model.

________________


Why This Separation Matters
The UserProfile must remain separate from User because:
                     * Authentication data must stay minimal and secure.

                     * Onboarding fields evolve frequently and should not affect login logic.

                     * Academic context may change over time without affecting identity.

                     * Future features (mentorship, peer groups, AI coaching modes) will rely heavily on profile-level attributes.

This separation maintains clean domain boundaries between:
                        * Identity (who the user is in the system)

                        * Academic intent (what the user is preparing for)

                        * Behavioral data (how the user studies)

                        * Performance data (how the user performs)

EXAM DEFINITION LAYER (REFERENCE STRUCTURE)
Quezia separates exam configuration from test instances.
An Exam represents a high-level assessment type (e.g., JEE, NEET, GATE).
An ExamBlueprint represents the canonical structure of that exam:
                                                         * version

                                                         * official subject composition

                                                         * default section layout

                                                         * default duration

                                                         * default marking scheme

                                                         * adaptive permissions

                                                         * effective dates

ExamBlueprintSection defines:
                                                            * subject identity

                                                            * sequence

                                                            * optional section time constraints

ExamRule defines:
                                                               * total time

                                                               * negative marking rules

                                                               * partial marking

                                                               * adaptive allowed

                                                               * effective date ranges

Blueprints are configuration artifacts.
They define boundaries and defaults.
They must never drive runtime dynamically.
They serve as:
                                                                  * Official template for system full mocks

                                                                  * Validation boundary for custom generation

                                                                  * Default configuration provider

________________


CRITICAL ARCHITECTURAL PRINCIPLE
Blueprint defines what is valid.
Test defines what is executed.
A test may:
                                                                     * Fully follow the official blueprint.

                                                                     * Partially follow it.

                                                                     * Override duration.

                                                                     * Override subject distribution.

                                                                     * Override question count.

                                                                     * Override marking (if allowed).

For example:
Official JEE blueprint may define:
                                                                        * 180 minutes

                                                                        * 90 questions

                                                                        * 30 per subject

But a user-generated test may define:
                                                                           * 90 minutes

                                                                           * 80 questions

                                                                           * 70 Physics

                                                                           * 10 Chemistry

This is valid practice behavior.
Therefore:
Blueprint is the canonical reference.
Test is the frozen execution artifact.

TEST GENERATION AND VERSIONING LAYER
Quezia supports regeneration and refinement. Therefore, it requires a TestThread abstraction.
TestThread:
                                                                              * id

                                                                              * examId

                                                                              * createdByUserId (nullable; null = system test)

                                                                              * originType (SYSTEM or GENERATED)

                                                                              * title

                                                                              * baseGenerationConfig (JSON snapshot)

                                                                              * createdAt

Each regeneration creates a new Test under the same thread.
Test:
                                                                                 * id

                                                                                 * threadId

                                                                                 * versionNumber

                                                                                 * examId

                                                                                 * blueprintReferenceId (nullable)

                                                                                 * durationSeconds

                                                                                 * totalQuestions

                                                                                 * totalMarks

                                                                                 * ruleSnapshot (JSON)

                                                                                 * sectionSnapshot (JSON)

                                                                                 * followsBlueprint (boolean)

                                                                                 * status (DRAFT, PUBLISHED, ARCHIVED)

                                                                                 * createdAt

When a test is created, all structural data must be snapshotted:
                                                                                    * Duration

                                                                                    * Marking rules

                                                                                    * Subject distribution

                                                                                    * Section breakdown

                                                                                    * Total marks

The test must never dynamically read the blueprint after creation.

________________


SYSTEM TESTS VS GENERATED TESTS
System tests:
                                                                                                                                                * Follow official blueprint

                                                                                                                                                * originType = SYSTEM

                                                                                                                                                * followsBlueprint = true

                                                                                                                                                * published to cohort

Generated tests:
                                                                                                                                                   * originType = GENERATED

                                                                                                                                                   * createdByUserId set

                                                                                                                                                   * may override structure

                                                                                                                                                   * followsBlueprint may be false

Retake behavior:
Retake creates a new TestAttempt.
Regeneration creates a new Test under the same TestThread.

                                                                                                                                                      * Flexible test generation configurations

                                                                                                                                                      * Snapshot-based immutable test definitions
